# ModelScope Image Gen MCP 产品与信息架构

## 文档状态

- 状态：已确认
- 前置文档：`00-rebuild-direction.md`
- 适用目标：重构后的 V1

本文定义 V1 的用户角色、核心工作流、工具能力边界、状态可见性和 Agent 使用体验。本文不定义最终字段类型、JSON Schema 或持久化结构；这些内容分别由领域、配置存储和 MCP 接口文档负责。

## 1. 产品定位

V1 是一个本地优先、ModelScope 专用、面向 Agent 的文生图 MCP 服务。

它将 ModelScope 的异步图像生成能力表达为可提交、可查询、可恢复、可安全落盘的本地任务工作流。Agent 不需要理解 ModelScope 原始响应、轮询状态或文件系统细节，只需要遵循服务返回的状态、等待建议和下一步动作。

## 2. 用户角色

### 2.1 Agent

Agent 是主要用户，负责：

- 提交文生图请求。
- 保存并使用本地 Job ID。
- 根据建议等待时间安排后续查询。
- 判断任务是否完成、失败或可以重试。
- 获取并消费最终图片列表。
- 在阻塞调用超时后转入异步工作流。
- 在上下文丢失或服务重启后查找并恢复最近任务。

Agent 不负责：

- 解释 ModelScope 原始状态。
- 拼接任意文件系统路径。
- 自行解析上游错误正文。
- 管理任务数据库。
- 决定 Token、输出根目录或隐私保留策略。

### 2.2 MCP Host

MCP Host 是程序化消费者，关注：

- 工具输入和结构化输出契约。
- 工具的只读性、网络访问和文件写入副作用。
- 调用超时、取消和错误语义。
- 模型可读内容与程序可读内容的一致性。
- 图片产物如何被用户、模型或后续工具消费。

### 2.3 本地操作者

本地操作者负责：

- 配置 ModelScope Token。
- 配置默认模型、任务状态位置和产物根目录。
- 控制日志等级、数据保留和清理策略。
- 查看诊断信息。
- 管理本地磁盘、权限和隐私数据。
- 显式运行真实 ModelScope 集成测试。

操作者配置不通过 MCP 工具暴露给 Agent。

## 3. 工具能力集合

V1 对外提供五项工具能力。

| 工具 | 定位 | 主要副作用 |
|---|---|---|
| `submit_image_generation` | 创建异步文生图任务 | 调用 ModelScope；创建本地任务记录 |
| `check_image_generation` | 检查并刷新单个任务状态 | 调用 ModelScope；更新本地任务记录 |
| `fetch_image_generation_result` | 下载、验证并保存已完成结果 | 网络下载；安全文件写入；更新产物状态 |
| `list_image_generations` | 查找和恢复本地任务 | 无上游访问；只读本地任务数据 |
| `generate_image` | 阻塞式便利入口 | 组合提交、轮询、下载和保存 |

这些名称和能力边界已经确认。参数、返回类型、工具 annotations 和精确错误结构将在 `05-mcp-interface-contract.md` 中定义。

### 3.1 命名原则

- 使用 `submit` 表示创建新的上游异步任务。
- 使用 `check` 而不是 `get_status`，因为调用会访问上游并更新本地状态。
- 使用 `fetch` 而不是 `get_result`，因为调用会下载远程内容并产生本地产物。
- 使用 `list` 表示只读发现，不隐式刷新上游状态。
- `generate_image` 是对完整工作流的便利编排，不是独立业务实现。

## 4. 默认异步工作流

异步工作流是 Agent 和支持后续调度的 MCP Host 的默认路径。

```text
submit_image_generation
    ↓
返回 job_id、当前状态、建议等待时间、下一步动作
    ↓
check_image_generation
    ├─ 未完成 → 返回下一次 check 和建议等待时间
    ├─ 失败   → 返回稳定错误、重试判断和可行建议
    └─ 成功   → 返回 fetch_image_generation_result
                    ↓
        fetch_image_generation_result
                    ↓
        下载、验证并安全保存零到多张远程结果
                    ↓
            返回 list[GeneratedImage]
```

### 4.1 提交阶段

`submit_image_generation`：

- 校验文生图请求。
- 创建本地 Job ID。
- 调用 ModelScope 创建上游任务。
- 将本地 Job ID 与上游 Task ID 关联并持久化。
- 返回足以继续工作流的信息。
- 不等待图片生成完成。
- 不下载任何图片。

只有在确认上游接受任务并返回可追踪标识后，提交才算成功。

### 4.2 检查阶段

`check_image_generation`：

- 每次调用最多执行一次上游状态查询。
- 不在服务器内部循环等待。
- 将 ModelScope 状态转换为稳定领域状态。
- 持久化状态变化和脱敏诊断信息。
- 返回建议等待时间和下一步动作。
- 不下载图片。

一次检查调用的网络超时或轮询预算耗尽不能把 Job 写成不可恢复的终态。只要没有上游失败证据，后续调用仍然可以继续刷新。

### 4.3 获取结果阶段

`fetch_image_generation_result`：

- 只处理上游结果已经就绪的任务。
- 支持一次任务包含 `list[GeneratedImage]`。
- 下载、验证并保存每张图片。
- 为每张图片独立记录下载和本地产物状态。
- 返回已经就绪的全部产物以及未成功产物的稳定错误信息。
- 重复调用应当幂等。

上游成功与所有本地产物可用是两个不同事实。下载或保存失败不能抹去上游已经成功的生成结果；调用方必须能够在后续再次执行 `fetch_image_generation_result`。

## 5. 阻塞式便利工作流

`generate_image` 面向简单脚本、测试和明确愿意等待的调用方。

```text
generate_image
    ↓
submit → 内部等待与检查 → fetch
    ↓
list[GeneratedImage]
```

必须遵守：

- 工具描述不得将其推荐为 Agent 默认路径。
- 内部复用与异步工具相同的应用用例、领域状态机和基础设施端口。
- 不允许维护独立的提交、轮询、下载或保存实现。
- 必须有明确的最大等待时间。
- 必须响应工具调用取消，不继续无界等待。
- 本地等待取消不等于上游任务取消。
- 等待超时时返回已经创建的 `job_id`、当前任务状态和异步下一步动作。
- 阻塞超时不能浪费或隐藏已经提交的上游任务。

如果 ModelScope 后续确认支持可靠的上游取消，应在独立决策中扩展领域和接口；不得把本地等待取消伪装成上游取消。

## 6. 任务发现与恢复

`list_image_generations` 用于让持久化任务真正可发现、可恢复。

典型场景：

- Agent 丢失或遗忘 `job_id`。
- MCP 服务进程重启。
- Host 会话中断后重新连接。
- 查找尚未完成的任务。
- 查找已经成功但尚未下载本地产物的任务。
- 检查最近失败任务的稳定错误摘要。

必须遵守：

- 默认只读取本地任务数据。
- 不为了列表展示而批量轮询上游任务。
- 支持按稳定领域状态过滤。
- 支持有界分页或数量限制。
- 默认按最近更新时间倒序。
- 返回每个任务的下一步动作。
- 默认不返回完整 prompt、签名 URL、原始错误正文或其他敏感大字段。
- 列表项是摘要，不等同于单个任务的完整状态结果。

V1 不提供通过 MCP 删除或批量清理任务的工具。保留与清理策略由操作者配置和后续存储契约负责，避免 Agent 获得不必要的破坏性能力。

## 7. 状态信息架构

### 7.1 Job 状态

Agent 主要看到稳定的领域状态，而不是 ModelScope 原始状态。

第一阶段的 Job 状态集合为：

```text
submitting
submitted
in_progress
succeeded
failed
```

含义：

- `submitting`：本地任务已经持久化，正在创建上游任务，但尚未获得可靠的上游 Task ID。
- `submitted`：上游已经接受任务，尚未观察到实际处理进度。
- `in_progress`：上游明确报告正在排队或处理。
- `succeeded`：上游明确报告生成成功，并返回至少一个结果引用。
- `failed`：上游明确报告失败，或上游成功响应违反不可恢复的结果契约。

`submitting` 用于覆盖“上游调用已经开始，但本地尚未获得或持久化 Task ID”的崩溃窗口。处于该状态的遗留任务需要在重启恢复时被识别为提交结果不确定，不能自动重复提交。

`timeout` 不是 Job 终态。超时描述某次调用、等待或网络行为，没有足够证据证明上游任务已经终止。

### 7.2 产物状态

本地产物状态独立于 Job 状态。初步信息架构使用：

```text
not_ready
pending_download
available
download_failed
```

它必须能够表达：

- 上游任务尚未成功，图片引用不可用。
- 上游已成功，但图片尚未下载。
- 部分或全部图片已经安全保存。
- 上游已成功，但某次下载、验证或保存失败。

精确状态机、部分成功规则以及是否需要更细的保存状态，由 `03-domain-model-and-behavior-map.md` 定义。

### 7.3 上游状态

ModelScope 原始状态只作为诊断事实保存和按需返回，不作为 Agent 控制流的主要依据。

未知上游状态不得静默映射为 `in_progress`。它必须触发明确的上游契约错误，同时保留足以诊断 API 变化的脱敏信息。

## 8. Agent 必须看到的信息

任务相关结果必须能够表达：

- 本地 `job_id`。
- 稳定 Job 状态。
- 当前工作流阶段。
- 是否为终态。
- 上游结果是否已就绪。
- 本地产物是否已就绪。
- 预期或已发现的图片数量。
- 创建时间和最近更新时间。
- 当前调用是否建议重试。
- 建议等待秒数。
- 下一步工具和必要参数。
- 最近一次稳定错误摘要。
- 必要时提供脱敏后的上游请求 ID。

Agent 不应默认看到：

- API Token。
- Authorization Header。
- 完整签名 URL。
- 完整 prompt。
- ModelScope 原始响应。
- Python 异常堆栈。
- 任务数据库内部位置。
- 不需要暴露的绝对内部路径。

列表摘要不返回完整 prompt。单任务详情中的 prompt 可见性由 MCP 契约最终定义；available 产物可以在 fetch 和单任务完整结果中返回解析后的绝对路径，但列表摘要不返回绝对路径。

## 9. 下一步动作

每个任务相关结果必须根据当前事实生成一致的下一步动作。

基本映射：

| 当前事实 | 下一步动作 |
|---|---|
| 已提交或处理中 | `check_image_generation` |
| 上游成功但产物未全部可用 | `fetch_image_generation_result` |
| 上游成功且所有产物可用 | 无必需动作 |
| 上游明确失败 | 无自动动作；返回重试判断与建议 |
| 阻塞等待超时 | `check_image_generation` |
| 单次检查网络失败但上游状态未知 | 重试 `check_image_generation` |
| 单次下载失败 | 重试 `fetch_image_generation_result` |

下一步动作是结构化契约，不应只存在于自然语言描述中。

## 10. 错误体验

错误信息必须帮助 Agent 回答三个问题：

```text
发生在哪个阶段？
是否值得重试？
下一步应该做什么？
```

错误至少区分：

- 输入校验失败。
- 服务配置缺失。
- 提交网络或 HTTP 失败。
- 上游拒绝或任务失败。
- 状态响应违反上游契约。
- 下载网络或内容失败。
- 图片验证失败。
- 本地保存失败。
- 持久化失败。
- 当前 Job 状态不允许该操作。
- 未预期内部错误。

错误字段、reason code、是否作为 MCP tool error 返回，以及结构化错误 envelope，将在 `05-mcp-interface-contract.md` 中定义。

## 11. 工具副作用与审批提示

工具元数据和描述必须准确表达副作用：

- `list_image_generations` 是本地只读操作。
- `check_image_generation` 会访问网络并更新本地状态，但不会生成新任务或写入图片文件。
- `submit_image_generation` 会调用外部服务并可能消耗上游额度。
- `fetch_image_generation_result` 会访问网络并写入本地文件。
- `generate_image` 会调用外部服务、等待任务并写入本地文件。

不能为了减少 Host 确认提示而把有副作用工具标记成只读或幂等。是否使用 MCP tool annotations 以及具体 annotations 值由 `05-mcp-interface-contract.md` 定义。

## 12. 信息层级

每次工具响应按照以下优先级组织：

1. 当前结果或状态。
2. 下一步动作。
3. 是否可重试及建议等待时间。
4. Job 和图片标识。
5. 必要诊断信息。
6. 低优先级元数据。

模型可读文本应简短总结结构化结果，不重复倾倒完整 JSON，也不隐藏只存在于文本中的关键控制信息。

## 13. V1 验收场景

产品工作流至少覆盖以下验收场景：

1. 提交任务，等待一次或多次检查后成功获取一张图片。
2. 一次任务产生多张图片，全部独立保存并返回。
3. 服务重启后根据 `job_id` 继续检查任务。
4. Agent 丢失 `job_id` 后通过列表找回最近未完成任务。
5. 阻塞生成超时后返回 `job_id`，并通过异步路径继续完成。
6. 上游成功但第一次下载失败，第二次 fetch 成功。
7. 多张图片中部分下载失败，成功产物不被丢弃或重复写入。
8. 上游明确失败，返回稳定错误和是否建议重新提交。
9. 单次状态查询网络失败，Job 不被错误写成失败或超时终态。
10. 未知上游状态被识别为契约错误，而不是无限等待。
11. 重复 fetch 已完成任务，不重复产生冲突文件。
12. 列表工具不批量访问 ModelScope，也不泄露完整敏感字段。
