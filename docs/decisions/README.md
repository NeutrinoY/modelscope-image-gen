# 项目决策记录

本目录保存重构基线之后，对产品、架构、契约、数据、安全和运维产生长期影响的新决策。它不是日常提交日志，也不用于记录可以由现有原则直接推出的普通实现细节。

## 与现有文档的关系

`docs/rebuild/00`–`07` 是 0.2 重构时已经确认的 V1 决策基线；`08-implementation-brief.md` 是完成该重构的实施交接。后续维护不应回写历史文档来伪造当时的认知。

新事实需要改变既有基线时，应在本目录建立新的决策记录，明确它取代或修订了哪些既有条目，再同步更新当前维护手册、代码、测试和用户材料。历史文档仍保留原貌，并通过新记录建立取代关系。

## 何时必须记录

以下变化通常需要决策记录：

- 增删、重命名公开 MCP Tool，或改变 wire schema、状态和错误语义；
- 改变默认异步工作流、本地优先定位或 Agent 下一步动作模型；
- 引入新的 Provider、图生图、图片编辑、取消能力、远程传输或多租户模式；
- 改变 Job/Artifact 状态机、事实归属、一致性或恢复语义；
- 改变数据库、迁移兼容策略、数据保留和删除语义；
- 改变 Artifact 路径控制、文件可见性或安全边界；
- 改变 MCP capability、transport、认证或部署信任模型；
- 替换核心运行时、异步模型、构建工具或质量工具链；
- 引入会长期改变层次责任或依赖方向的架构方案。

局部缺陷修复、私有重命名、现有边界内的重构和由既有决策唯一确定的实现通常不需要单独记录。

## 文件名与状态

文件使用递增编号和简短英文名称：

```text
0001-short-decision-name.md
0002-another-decision.md
```

状态使用：

- `Proposed`：讨论中，不能作为已接受约束；
- `Experimental`：允许受控验证，尚未形成长期方向；
- `Accepted`：已经确认并应被贯彻；
- `Superseded`：被另一项决策明确取代；
- `Rejected`：已评估但不采用，保留排除理由。

## 模板

```markdown
# NNNN：决策标题

- 状态：Proposed
- 日期：YYYY-MM-DD
- 决策者：
- 取代：无
- 被取代：无

## 背景

什么新事实、问题或目标触发了决策。

## 必须保持的约束

哪些产品语义、数据事实、安全边界和兼容要求不能丢失。

## 方案

考虑过的主要方案及其真实权衡。

## 决策

最终选择什么，以及为什么。

## 后果

正面收益、负面代价、兼容性、数据、安全、运维和维护影响。

## 实施与迁移

需要修改的系统范围、迁移顺序、回滚或失败策略。

## 验证

怎样证明该决策正确落实；哪些证据只能由 live、Host 或发布环境提供。

## 永久知识同步

需要更新的维护文档、契约、用户材料和自动化护栏。

## 重新评估条件

出现什么新事实时应重新讨论。
```

## 接受规则

只有当影响已经说明、决策者明确确认、状态改为 `Accepted`，并且取代关系清楚时，记录才构成新的项目级约束。较新的随口建议、未合并实验或单独的代码偏移不会自动取代既有决策。
